## Deploy in One Command

The fastest way to get your code live is with `bool shipit` — no account or API key required.

<Steps>
  <Step title="Navigate to your project">
    Open your terminal and navigate to the directory containing your project files:

    ```bash
    cd my-project
    ```
  </Step>

  <Step title="Ship it!">
    Run the `shipit` command to create an anonymous Bool and deploy your files:

    ```bash
    bool shipit
    ```

    <Note>
      By default, `bool shipit` deploys the current directory (`.`). You can specify a different directory:
      ```bash
      bool shipit ./src
      ```
    </Note>

    You'll see output like this:

    ```bash
    https://abc123xyz.bool01.com
    ✔ Shipped "my-project" (12 files)
      https://abc123xyz.bool01.com
      slug: abc123xyz
    ```
  </Step>

  <Step title="Visit your live site">
    Your project is now live! Open the URL displayed in the output to see your deployed site.

    The CLI automatically creates a `.bool/config` file in your project directory to track the deployment.
  </Step>
</Steps>

## Update Your Deployment

Once you've deployed with `bool shipit`, subsequent runs from the same directory automatically update your existing deployment:

```bash
# Make some changes to your code
echo "console.log('Updated!')" >> script.js

# Ship the updates
bool shipit
```

Output:
```bash
https://abc123xyz.bool01.com
✔ Updated v2 (12 files)
  https://abc123xyz.bool01.com
  slug: abc123xyz
```

<Info>
  The `.bool/config` file stores your Bool's `slug` and `name`. This allows the CLI to update the correct deployment without you needing to specify the slug each time.
</Info>

## Add a Commit Message

When updating an existing deployment, you can add a commit message to track changes:

```bash
bool shipit -m "Fixed navigation bug"
```

This message will be visible in your version history.

## What Gets Deployed?

Bool CLI automatically handles file filtering for you:

### Included Files
- All text-based source files (`.js`, `.html`, `.css`, `.json`, etc.)
- Files are read recursively from your project directory

### Excluded by Default
- `.git` - Git repository data
- `node_modules` - Node.js dependencies
- `__pycache__` - Python cache files
- `.DS_Store` - macOS system files
- `.bool` - Bool CLI configuration
- Binary files (images, videos, fonts, archives, etc.)

### Custom Exclusions

Create a `.boolignore` file in your project root to exclude additional files using gitignore syntax:

```text .boolignore
# Exclude test files
*.test.js
__tests__/

# Exclude environment files
.env*

# Exclude documentation
docs/
*.md
```

<Warning>
  Binary files like images and fonts are automatically detected and skipped during the initial deployment. They are uploaded separately in the background and may take a few moments to appear on your live site.
</Warning>

## Advanced Options

### Update a Specific Bool

You can update an existing Bool by providing its slug:

```bash
bool shipit --slug abc123xyz
```

### Custom Bool Name

By default, the Bool is named after your directory. Override this with the `--name` option:

```bash
bool shipit --name "My Awesome Project"
```

### Skip File Uploads

To skip uploading binary files (useful for testing):

```bash
bool shipit --no-upload
```

## Understanding the Config File

After your first deployment, Bool CLI creates a `.bool/config` file:

```json .bool/config
{
  "slug": "abc123xyz",
  "name": "my-project",
  "secret": "aBcDeFgHiJkLmNoPqRsTuVwXyZ123456"
}
```

- **slug**: Unique identifier for your Bool (used in the URL)
- **name**: Display name for your project
- **secret**: Authentication token for updating this anonymous Bool

<Warning>
  Add `.bool/` to your `.gitignore` to avoid committing the secret to version control:
  ```text .gitignore
  .bool/
  ```
</Warning>
